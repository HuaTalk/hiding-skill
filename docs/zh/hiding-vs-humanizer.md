# /hiding 与 humanizer：内容泄露与文体指纹

两个工具都宣称同一个目标——"让 AI 生成的东西看起来像人写的"。但它们解决的是两个不同层面的问题，几乎不重叠。本文对比两者的定位差异，并说明 `/hiding` 的能力边界。

对比对象是 [blader/humanizer](https://github.com/blader/humanizer) 及其衍生版本（如中文译版 humanizer-zh），基于 Wikipedia [Signs of AI writing](https://en.wikipedia.org/wiki/Wikipedia:Signs_of_AI_writing) 页面整理的 AI 写作特征指南。

## 一句话定位

> **/hiding 管"内容说漏"，humanizer 管"腔调露馅"。**

- `/hiding` 处理**语义泄露**：文件里出现了不该出现的内容——凭据、规则引用（"按照 CLAUDE.md 的约定..."）、思维推导过程、AI 自我指涉。这些是人类 reviewer 一眼就能看出的内容级痕迹。
- humanizer 处理**文体指纹**：内容本身没问题，但表达方式带有 AI 的统计特征——破折号过度使用、三段式列举、"此外/彰显/格局"等高频 AI 词汇、以 -ing 结尾的肤浅分析、宣传腔。这些是 AI 检测器（GPTZero 类）赖以工作的信号。

一个文件可以通过 `/hiding` 清理后不含任何语义泄露，但文风仍然一眼 AI；也可以经 humanizer 改写后文风自然，但仍然在注释里写着"根据 skill 指示..."。两者叠加才构成完整的"像人写的"。

## 维度对比

| 维度 | /hiding | humanizer |
|------|---------|-----------|
| 解决什么 | 语义泄露：说了不该说的 | 文体指纹：说话方式像 AI |
| 操作对象 | 文件（代码、配置、Markdown、文档） | 文本（文章、prose） |
| 操作方式 | **只删不改写**——代码逻辑永不改动 | **重写**——重构句式、变换节奏、注入个人语气 |
| 输出行为 | 静默执行：运行后无人能看出运行过 | 高调输出：改写文 + 变更总结（部分版本附评分） |
| 安全层 | Pattern S 凭据剥离 + 强制轮换警告 | 无——完全不涉及 secrets |
| 交互控制 | HITL、`--dry-run`、三种输出模式、`--subagent` | 无 flag、无流程控制，纯 pattern 指南 |
| 结构保障 | 清理后用真实 parser（JSON/YAML/XML）校验 | 无 |
| pattern 依据 | 自定义原则（判断式，明确反对关键词 grep） | WikiProject AI Cleanup 实证观察（词表式，接近可 grep） |

## 唯一的重叠区

`/hiding` 的 Pattern A（AI 自我指涉）与 humanizer 的"协作交流痕迹"、"谄媚语气"两类 pattern 覆盖同一批内容：

> "Here's the result:"、"希望这对您有帮助！"、"Great question!"、"Let me know if..."

处理方式不同：`/hiding` 直接删除（这些句子在文件中没有存在价值）；humanizer 改写为正文（它假设周围还有需要保留的 prose）。对代码注释和配置文件，删除是正确动作；对文章正文，改写是正确动作。重叠区的分工其实也是按操作对象划分的。

## 为什么 /hiding 不做文体人性化

`/hiding` 的核心原则之一是**只删不改写**（代码逻辑永不改动，只剥离注释与 prose）。文体人性化必然要求改写——重构句式、替换词汇、调整节奏。两者不可兼得：

1. **改写引入风险**。删除一行注释不会改变语义；改写一段文档可能改变含义。`/hiding` 的结构校验（Step 4）建立在"删除是低风险操作"的前提上，改写会破坏这个前提。
2. **改写违背静默原则**。humanizer 类工具需要输出变更总结让用户确认改写没有失真——这本身就是一种"高调"。`/hiding` 的静默执行依赖操作足够保守，保守到不需要确认。
3. **文体问题有专门工具**。与其在 `/hiding` 里内置一个降级版 humanizer，不如保持边界清晰，推荐链式使用。

## 推荐用法：链式清理

对含大量 prose 的 Markdown 文档，完整流程是两步：

```
/hiding doc.md          # 第一步：剥离语义泄露（凭据、规则引用、推导过程、自指涉）
<humanizer> doc.md      # 第二步：改写残余的 AI 文体（破折号、三段式、AI 词汇）
```

顺序重要：先 `/hiding` 后 humanizer。反过来的话，humanizer 可能把"我们按照 CLAUDE.md 的约定选择了 X"改写成更自然的"按团队约定选择了 X"——文体变好了，但规则引用（Pattern R）还在，且变得更难识别。

对纯代码文件，通常只需要 `/hiding`——代码注释被剥离后剩下的是可执行代码，不存在文体问题。

## 能力边界声明

`/hiding` 的承诺"清理后文件读起来像人类编写"应准确理解为：

- **覆盖**：内容级痕迹。清理后文件中不存在暴露 AI 来源的语义内容。
- **不覆盖**：文体级特征。Markdown 文档的行文风格、词汇选择、句式节奏不在 `/hiding` 的处理范围内。若文档需要通过 AI 文体检测或面向对文风敏感的读者，应在 `/hiding` 之后叠加 humanizer 类工具。

## 互相借鉴的方向

**`/hiding` 可以向 humanizer 借鉴：**

- pattern 依据的外部背书。humanizer 引用 Wikipedia 的实证观察数据；`/hiding` 的五 pattern 是自定义原则，补充外部参照（AI 检测研究、真实泄露案例）能增强可信度。

**humanizer 类工具可以向 `/hiding` 借鉴：**

- 安全层。humanizer 完全不检查 secrets——一篇待"人性化"的文章里若含内部 URL 或 token，会被原样保留甚至改写得更隐蔽。
- 工程化。flag 体系、dry-run 预览、输出模式、版本管理与 CI，humanizer 类工具普遍缺失。
- 结构校验。改写比删除风险更高，却没有任何 post-hoc 验证。

## 结论

| | 内容干净 | 内容泄露 |
|---|---|---|
| **文风自然** | 目标状态 | 只跑了 humanizer |
| **文风 AI** | 只跑了 /hiding | 原始输出 |

`/hiding` 与 humanizer 不是竞品，是相邻两层防线。`/hiding` 选择把边界画在"只删不改写"上，代价是放弃文体层，收益是静默、安全、结构可验证。文体层交给专门工具，链式使用。
