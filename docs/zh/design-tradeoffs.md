# 设计取舍：事后清理而非实时约束

## 问题

AI 编码助手生成的文件常携带"泄露痕迹"——AI 自指涉（"Here's the result:"）、推导链条（"we chose X because Y..."）、规则引用（"按照 CLAUDE.md 的约定..."）等。这些内容一旦提交到仓库，暴露了文件的 AI 来源，降低专业度，甚至泄露安全信息。

面对这个问题，有两种策略：

- **策略 A（实时约束）**：在每会话中注入规则，要求模型在生成时主动避免泄露。
- **策略 B（事后清理）**：模型自由生成，用户需要时运行清理命令（`/hiding`）统一剥离。

`/hiding` 选择了策略 B。本文解释为什么。

## 实时约束的隐性代价

### 1. 指令遵循与推理能力的损耗

以始终开启的规则文件为例（~400 tokens），注入到每会话的上下文中：

```markdown
Before committing, pushing, or sharing files, scan them for AI-generated artifacts.
- Don't say "As an AI", "I think", "Here's the result"
- Don't include reasoning trails in comments
- Don't reference CLAUDE.md or skill instructions
...
```

这 ~400 tokens 不是"提醒"——是**约束**。模型每次生成输出时，都需要在约束条件间做额外判断：

- "用户让我解释设计思路，规则却说不要写推导过程——我怎么表达？"
- "这是一个 code review 场景，说'I think'是自然的——会被标记吗？"
- "TODO 标记是团队约定还是 AI 自言自语？"

每一次判断都消耗注意力。当模型在自我审查时，它不在推理。

### 2. 防线不完整：规则注入不能替代清理

即使注入了规则，模型仍然会泄露：

- AI 自指涉太自然了——"Let me..."、"Here's..." 是人类和 AI 都用的表达，模型做不到 100% 准确区分
- 推理链条有时是正确的输出——用户确实需要知道"为什么选 X 而不是 Y"，但格式上暴露了 AI 思考过程
- 约束之间会冲突——"保持简洁"和"解释清楚"和"不写 AI 痕迹"三者互斥

结果：无论有没有规则注入，用户最终都要跑一遍 `/hiding`。规则注入省不掉清理步骤，却付出了 tokens 和推理质量的代价。

### 3. 输出质量的悖论

最微妙的问题：实时约束会让模型输出变得"不自然"——刻意回避某些短语、句式生硬、回避合理的解释。这种不自然本身就是一种 AI 痕迹。

一个经过实时约束的模型，输出看起来像**被限制过的 AI**。而一个自由发挥后经 `/hiding` 处理的文件，读起来像**人类原本写的那样**。

这正是 `/hiding` 静默执行哲学成立的前提：

> after `/hiding` runs, no one should be able to tell it ran

如果模型在生成时就已经被规则扭曲了，那"静默"没有意义——文件从一开始就带着"被约束过"的痕迹。

## 事后清理的优势

### 1. 模型能力不受损

模型自由发挥，用最自然的语言表达，token 全部用于推理和内容质量。`/hiding` 在最后一步统一处理——该删的删，该改的改，不给模型增加实时负担。

### 2. 清理质量更高

事后清理不受实时生成的压力——可以反复审视每一行、每一个注释、每一个键值对。Step 0-4 的层层把关（校验 → 删除检查 → 剥离机密 → 剥离风格泄露 → 验证结构）比实时自检更彻底。

尤其是 Step 1（文件级删除检查）：当整个文件 80%+ 内容都是 AI 推导时，逐行清理没有意义——直接删除整个文件。这种判断只有在文件完成后才能做出。

### 3. 用户可选择、可控制

不是每个场景都需要清理。探索性编码、内部讨论、临时文件——用户可以接受带着 AI 痕迹。只在提交、推送、分享前运行 `/hiding`。

HITL（Human-in-the-Loop）模式更进一步：无参数运行 `/hiding` 时，它分析会话中所有文件，列出泄露候选项，让用户选择处理哪些。控制权始终在用户手中。

### 4. 哲学一致性

`/hiding` 的静默执行哲学要求**操作后不可追踪**。这个要求对策略 A 是矛盾的（实时规则本身就是一种"痕迹"——模型行为被改变了），但对策略 B 是自洽的（清理是事后操作，文件发布前已完成，交付物无痕迹）。

## 代价与权衡

选择了事后清理，意味着：

- **用户需要记住运行命令**——这是最大的劣势。实时规则不需要用户操作。对策：将 `/hiding` 融入提交流程习惯，在 `git commit` 前手动运行。不注入会话提醒——任何提醒本质上都是约束，与事后清理哲学矛盾。

- **不能替代安全意识**——机密泄露（Pattern S）在事后清理时已造成损失。用户不应依赖任何事后工具来管理凭证——密钥管理应使用环境变量和密钥管理器。`/hiding` 的 Pattern S 清理是最后一道防线，不是第一道。

- **大文件有性能开销**——Step 0 限制文件大小。超大文件需要分批处理。

## 总结

| 维度 | 实时约束 | 事后清理 |
|------|---------|---------|
| 推理质量 | 受影响 | 不受影响 |
| 输出自然度 | 不自然（带着约束痕迹） | 自然（自由生成后处理） |
| 清理彻底性 | 不完整（模型做不到 100%） | 更彻底（逐行审查） |
| 用户控制 | 被动（每会话生效） | 主动（需要时调用） |
| 是否需要额外操作 | 否 | 是 |
| 静默哲学一致性 | 不一致（行为改变可追踪） | 一致（交付前已清理完毕） |

`/hiding` 选择了事后清理，因为它的目标不是改变模型的行为，而是保护交付物的质量。好的工具不该教模型怎么说话——它应该让模型自由说话，然后在需要的时候，无声地擦掉不该留下的痕迹。
