## 📖 用户故事 — 安装与上手

### US-01: 一句话安装

> **作为** 刚听说这个工具的小张
> **我想要** 用一条命令装上 `/hiding`
> **以便** 不需要研究复杂配置就能开始用

```bash
npx skills add HuaTalk/hiding-skill
```

**当前状态：** ✅ 已满足。一条命令完成安装，覆盖 70+ 代理。

**摩擦点：** 安装后需要**重启** Claude Code。

---

### US-02: 第一次使用，确认工具有效

> **作为** 刚安装完的小张
> **我想要** 试一下这个命令到底做了什么
> **以便** 确认它真的有用，而不是装了就不管

```
/hiding
```

**当前状态：** ⚠️ 有摩擦。如果当前会话没有创建/修改过文件，HITL 模式返回"零发现 → 静默"。
**用户感知：** 命令似乎什么都没做。是不是没装好？是不是不生效？

**建议：** 零发现时给出简短确认："未在此会话中发现 AI 泄露痕迹。"（HITL 模式本身就是例外，可以出声）

---

### US-03: 用示例文件看效果

> **作为** 想快速了解效果的小张
> **我想要** 一个"前后对比"的演示
> **以便** 直观看到工具删除了什么、保留了什么

**当前状态：** ❌ 不支持。README 没有 before/after 示例，也没有 `/hiding demo` 命令。

**建议：** README 加一个 3 行 before/after 示例；未来考虑 `/hiding demo` 生成一个带泄露痕迹的临时文件并清理。

## 🧹 用户故事 — 单文件清理（File Mode）

### US-04: 策略性清理一个 AI 生成的代码文件

> **作为** 刚用 Claude 生成了一个 `UserService.java` 的小张
> **我想要** 把文件里的 AI 痕迹清掉
> **以便** 后续读者无法从文件内容推断 AI 参与和内部约束

```
/hiding --files src/main/java/com/example/UserService.java
```

**理想体验：**
- 命令执行后，静默完成
- 文件里的 `// As an AI, I'll use...`、`// I think this approach...` 被移除
- 纯代码逻辑不变
- `git diff` 只看到注释被精简，函数体完好

**当前状态：** ✅ 核心路径。执行链：Step 0 校验 → Step 1 删除检查 → Step 2 剥离机密 → Step 3 剥离风格泄露 → Step 4 验证结构。

**摩擦点：**
- 静默结果无法确认是否真的清理了（新用户焦虑）
- Step 1 的 20% 阈值判断依赖 LLM 估算，可能不一致

---

### US-05: 清理前先看会清理什么

> **作为** 不想"盲操作"的阿明
> **我想要** 先预览哪些内容会被标记为泄露
> **以便** 确认不会误删重要内容

**当前状态：** ❌ 不支持。没有 `/hiding --dry-run` 或 `/hiding --preview` 模式。

**建议：** `--dry-run` 模式：输出会删除的内容但不修改文件。这是弥补"静默信任赤字"最直接的方案。

---

### US-06: 清理配置文件中的 API 密钥

> **作为** 发现 `application.yml` 里有一个测试用的 OpenAI API Key 的 Lisa
> **我想要** 把它移除
> **以便** 这个 key 不会出现在 git 历史里

```
/hiding --files config/application.yml
```

**理想体验：**
- Key 被移除
- 如果 key 已经 git 提交过，工具应**警告**需要去轮换凭证

**当前状态：** ⚠️ 核心问题。静默哲学和凭证安全之间存在矛盾：
- 工具会剥离凭证
- 但**不会告诉用户**凭证曾被剥离
- 用户不知道需要去轮换被泄露的 key

**这是上一轮分析中标记为 Critical 的发现。** 凭证剥离不能静默。

---

### US-07: 清理 Markdown 设计文档

> **作为** 用 AI 生成了一份技术方案文档的 老王
> **我想要** 删掉"调研发现...""经过分析...""最终选择 X 因为 Y..."这类推导过程
> **以便** 文档读起来像一份纯粹的决策记录，而不是 AI 的思考笔记

```
/hiding --files docs/design-proposal.md
```

**理想体验：**
- "选择了 Redis 而不是 Memcached，因为..." → 保留（这是业务决策，读者需要知道）
- "我调研了三种方案，首先排除了..." → 删除（这是推导过程）
- "根据团队规范 CLAUDE.md..." → 删除（读者没有 CLAUDE.md）

**当前状态：** ✅ 支持，但 Step 1 可能触发文件级删除（如果 80%+ 是推导过程）。触发 HITL，会询问用户是否删除整个文件。

**摩擦点：** AI-facing rationale/guardrails（面向 AI 的理由/护栏）和思考过程痕迹边界模糊，模型可能判断不一致。

---

### US-08: 处理被标记为"应删除"的文件

> **作为** 对一份 95% 都是 AI 思考记录的文件运行 `/hiding` 的小张
> **我想要** 被告知这个文件基本全是 AI 推导
> **以便** 我可以决定是删掉它还是重新写

**触发条件：** Step 1 — 移除思考过程痕迹后剩余实质性内容 < 20%。

**当前状态：** ✅ 支持。会弹出 AskUserQuestion 询问是否删除。

**摩擦点：** "实质性内容"的定义模糊。如果模型误判，可能建议删除一个本来有保留价值的文件。

---

## 🔍 用户故事 — 会话感知检查（HITL Mode）

### US-09: 下班前检查今天的 AI 痕迹

> **作为** 用 AI 辅助写了一整天代码的小张
> **我想要** 在下班前检查今天所有修改过的文件
> **以便** 不把 AI 痕迹、内部约束和过程线索留给后续使用者

```
/hiding
```

**理想体验：**
1. 工具扫描当前会话中所有创建/修改过的文件
2. 列出发现：
   - **Tier 1:** `research-notes.md` — 建议删除（纯 AI 思考过程）
   - **Tier 2:** `UserService.java` — 3 处 AI 自指涉，1 处规则引用
   - **Tier 3:** "今天讨论过 'Project Falcon' 这个内部代号 — 检查 docs/ 下的文件"
3. 用户勾选要处理的项
4. 执行，静默完成

**当前状态：** ⚠️ 部分支持。
- 会话范围未明确定义（"回顾整个对话"到底看多少？）
- HITL 发现用**单个** AskUserQuestion 呈现，选项最多 4 个。如果发现很多，放不下。
- 执行后"回归静默"——用户看不到确认。

---

### US-10: 只关心有没有泄密，不管风格

> **作为** 安全工程师 Chen
> **我想要** 扫描会话中所有文件是否有 API key、token、密码
> **以便** 在提交前挡住最危险的东西

**当前状态：** ❌ 不支持。HITL 把所有模式混在一起。

**建议：** `/hiding` 的 HITL 发现应该把凭据与密钥（安全问题）**单独置顶**，和其他模式（风格/质量）分开。安全发现必须突出显示，不能和"2 处 AI 自指涉"放在同一个 Tier。

## 🎯 用户故事 — 按描述清理（Description Mode）

### US-11: 全局隐藏 mock 数据标记

> **作为** 项目中到处写了 `// mock data for testing` 的小张
> **我想要** 把所有文件里涉及 mock 数据的痕迹清理掉
> **以便** 交付给客户时不会暴露测试数据

```
/hiding "mock data"
```

**理想体验：** 搜索上下文中所有匹配"mock data"描述的文件，应用相关隐藏模式，静默执行。

**当前状态：** ⚠️ 支持但模糊。
- "描述匹配"的范围未定义——是扫描哪些文件？工作区所有文件？还是仅会话中涉及的文件？
- 如果匹配到 30 个文件，全部静默修改——用户完全不知道改了哪些。
- "mock data" 也可能误匹配到正常的业务注释

**建议：** 描述模式应该先列出匹配的文件和预估影响范围，让用户确认后再执行。类似于 HITL 模式但按描述过滤。

---

### US-12: 全局隐藏内部代号

> **作为** 项目里有一个还没公开的内部代号"Project Nebula"的 阿明
> **我想要** 把所有文件和文档里提到"Project Nebula"的地方清理掉
> **以便** 对外发布前不暴露内部信息

```
/hiding "Project Nebula"
```

**当前状态：** ⚠️ 与 US-11 相同的问题——范围未定义，静默修改大量文件无反馈。

## 🔄 用户故事 — 工作流集成

### US-13: Git 提交时不忘记运行

> **作为** 每次 commit 前想自动提醒自己运行 `/hiding` 的小张
> **我想要** 一个 git hook 或者自动提醒
> **以便** 不会因为忘记运行而把 AI 痕迹提交上去

**当前状态：** ❌ 不支持。设计哲学禁止注入会话提醒。

**用户的变通方案：**
```bash
alias gc='echo "记得先跑 /hiding" && git commit'
```

**建议：** 在 README 中发布一个**推荐的但不自动安装**的 pre-commit hook 模板。用户主动选择安装，不违背静默哲学。

---

### US-14: CI 流水线里自动扫描

> **作为** 负责团队代码质量的 Lisa
> **我想要** 在 PR 提交时自动扫描是否有 AI 泄露
> **以便** 不依赖每个开发者都记得手动运行

```yaml
# .github/workflows/hiding-check.yml
name: Hiding Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npx hiding --ci  # 假设有这个命令
```

**当前状态：** ❌ 不支持。没有 CLI 工具，无法在 CI 中运行。

**建议：** 这是 `npx hiding` 独立 CLI 工具的核心使用场景。CI 模式下不需要静默——应该产生报告。

---

### US-15: 团队统一配置黑名单

> **作为** 想给团队统一配置哪些模式需要强制清理的 Lisa
> **我想要** 一个 `.hidingrc` 或配置文件
> **以便** 团队可以定制：只强制检查凭据与密钥，其他模式可选

**当前状态：** ❌ 不支持。没有任何配置机制。

**建议：** 未来考虑一个 `.hidingrc.json`：

## 🎯 关键洞察

1. **核心路径（US-04）已经很好了。** 单文件清理是产品最成熟的路径，设计合理。

2. **安全路径（US-06）有一个致命漏洞。** 凭证剥离后不告知用户，可能导致已泄露的凭证不被轮换。这是唯一要打破静默哲学的硬性理由。

3. **HITL 路径（US-09）概念好但执行不完整。** 会话感知分析是差异化的功能，但"会话范围"和"发现呈现"都需要更精确的定义。

4. **预览路径（US-05）是最大需求缺口。** Dry-run（试运行）是弥补"静默信任赤字"最简单有效的方法，多个用户故事都指向这个需求。

5. **自动化路径（US-13/14）决定了 tool → product 的跨越。** 当前 `/hiding` 是一个开发者的个人工具；加上 git hook 和 CI 集成后，它变成了一个团队工具。这是从 v0.5 到 v1.0 的质变。
