# 竞品格局：清理代码中 AI 痕迹的工具调研

> 快照日期：2026-07-16。该品类演进极快——下列多个竞品发布于本次调研前数月（甚至当天）。引用前请复核。

## 调研方法

对 agent skill 生态（skills.sh / npx skills、Claude Code 插件市场、GitHub 上的 SKILL.md 仓库）、独立 CLI 工具、humanizer 类文体改写工具做了 deep-research 扫描。共识别 10 个邻近竞品；以下每条结论均经 3 票对抗验证，核实到一手来源（README / 源码级 grep）。

## 核心问题

是否存在别的工具做 `/hiding` 做的事——事后清理文件中的内容级 AI 泄露，且覆盖全部五类内容？

| 模式 | `/hiding` 的目标 |
|------|------------------|
| 凭据与密钥 | 凭证、密钥、token + 强制轮换警告 |
| 未共享规则引用 | CLAUDE.md / skill 指令 / 团队约定引用 |
| 面向 AI 的理由/护栏 | 面向 AI 的约束推理（"不能用 X 因为团队要求 Y"） |
| AI 自我指涉 | "作为 AI…"、"我认为…"、"以下是结果：" |
| 思考过程痕迹 | 逐步推理、带日期的记录、调研痕迹 |

**答案：不存在。** 无任何竞品覆盖全部五维。未共享规则引用在所有竞品中覆盖为零。

## 结论

1. **未共享规则引用剥离是 `/hiding` 最深的护城河**——10/10 已核实竞品全部缺失。定位宣传中值得突出。
2. **机密 + 强制轮换警告的组合独有。** papertowel 只检测密钥，不给轮换指引。
3. **品类真实且正在升温**——undercover（2026-04）、papertowel（2026-04）、aislop v0.13.1（2026-06）、Hanfani Guard（2026-07-16）均为近期产物。需求已被验证；差异化窗口就在当下。
4. **「内容级泄露 vs 文体指纹」的区分成立**（[docs/zh/hiding-vs-humanizer.md](hiding-vs-humanizer.md)）。humanizer 类工具（ai-humanizer、papertowel 的文体检测器）在分界线的另一侧。
5. **潜在 roadmap 缺口**：Dogma 的 `sanitize-git` 能清理已提交历史；`/hiding` 只扫描未提交文件，对已提交凭证仅建议轮换。是否加历史改写是一个需要慎重决策的哲学问题（破坏性操作、需 force-push），不是疏漏——但值得留下一个成文决定。

## 局限说明
