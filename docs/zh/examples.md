# /hiding 示例

这些示例用于说明预期结果。检测依赖上下文判断：`/hiding` 会判断内容对于只能看到当前文件的读者是否仍然有用，而不是匹配固定关键词列表。

## 1. 临时会话记录

```python
# 清理前
# 会话记录：第一版返回 tuple；检查调用方后改为 dict。
def get_user() -> dict[str, str]:
    ...

# 清理后
def get_user() -> dict[str, str]:
    ...
```

该注释记录了本次会话的实现过程，不是维护代码所需的信息。

## 2. 私有规则引用

```markdown
清理前：

> 会话记录：比较过 REST、gRPC 和 GraphQL。

```

保留长期有效的决策，移除读者无法获得的规则引用和临时比较过程。

## 3. 泄露的写作指令

```markdown
清理前：

> 写作说明：面向没有技术背景的读者。
> 根据 Skill 指令，使用类比并避免公式。

语言模型会预测文本，可以把它理解为规模更大的联想输入。

清理后：
```

关于如何编写文档的指令不应出现在文档正文中。

## 4. 配置文件中的过程记录

```yaml

# 临时计划：拉取镜像、安装依赖、测试、构建、上传。

  - run: ./ci.sh

# 清理后
steps:
  - run: ./ci.sh
```

工作流本身已经足够清楚，提示词服从说明和临时计划对读者没有帮助。

## 6. 凭据处理

下面的值仅用于说明，不是可用凭据。

```yaml
# 清理前
service:
  api_key: "example-secret-value"

# 可能的格式安全候选版本
service:
  api_key: "<REDACTED>"
```

只有在配置结构保持有效时，才允许使用格式安全的占位符。可执行代码中的凭据不会被静默修改；`/hiding` 会报告位置并要求人工审查。

只要发现凭据，包括在 `--dry-run` 中发现，`/hiding` 都会提示可能需要轮换凭据。报告中绝不会显示发现的原始值。

## 7. 用户指定目标

```text
/hiding "internal project name" --files release-notes.md --dry-run
```

目标按语义解释。匹配的 prose 可以被移除，但标识符、可执行代码或影响行为的配置值只会被报告，不会被修改。

## 验证

生成清理候选版本后，`/hiding` 会重新读取文件，并在可用时使用解析器验证 JSON、YAML 或 XML。若结构验证失败，候选版本会被丢弃，原文件保持不变。

对于重要文件，应先运行 `--dry-run`，清理后再运行宿主项目自己的测试和专用密钥扫描器。
